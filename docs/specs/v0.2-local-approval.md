# PermitRail v0.2: local human approval

Status: draft spec, build target for v0.2.

## Goal

Turn PermitRail from "good infrastructure" into "usable today" by adding a local
human-approval flow for sensitive agent tool calls routed through PermitRail. A
developer runs their agent, a risky tool call pauses, a local web page shows the
exact action, a human approves or denies, PermitRail signs a proof, the tool runs
once, and a receipt is written.

Positioning, use this wording: PermitRail adds human approval to sensitive agent
tool calls routed through PermitRail. It does not control tools an agent calls
directly outside the gateway.

## Scope

In:
- A local approval server (zero dependency, binds to 127.0.0.1 only).
- A local approval page (dark console aesthetic, reuse the sandbox styling).
- An await helper so agent code can wait for the human decision.
- A runnable demo agent script.
- Docs: local vs production, with an explicit caveat.

Out (v0.3 and later): Slack, email, mobile, hosted cloud, database persistence,
team roles, accounts, multi-user auth.

## Package

`@permitrail/local-approval`. ESM, zero runtime dependencies (Node `http` plus
Web Crypto via `@permitrail/core`). Reuses `LocalApprovalProvider` from
`@permitrail/provider-local`. No new crypto.

Family naming rule:
- `provider-*` (provider-local, provider-webhook): low-level approval provider implementations.
- `local-approval`: the complete local approval experience (server, page, provider, and `waitForProof`).

README install line:
`npm install @permitrail/core @permitrail/mcp-gateway @permitrail/local-approval`

## Public API

```ts
import { startLocalApproval } from '@permitrail/local-approval';

const approval = await startLocalApproval({
  port: 4677,            // default 4677, localhost only
  title: 'Agent approvals',
});

// approval.provider      ProofProvider the gateway uses (signs on approve)
// approval.publicKeyPem  trusted key for the gateway
// approval.url           'http://127.0.0.1:4677'
// approval.waitForProof(challengeId, { timeoutMs? }): Promise<SignedEnvelope<ProofPayload>>
//                        resolves when approved, rejects on deny or expiry
// approval.stop(): Promise<void>
```

Wire into the existing gateway, no new gateway API:

```ts
const approval = await startLocalApproval();
const gateway = new PermitRailGateway({
  policy,
  provider: approval.provider,
  trustedProofKeys: [approval.publicKeyPem],
  receiptKeyPair, // generate once and persist
});

const decision = await gateway.authorize(action);
if (decision.outcome === 'require_proof' && decision.challenge) {
  console.log(`Approve this action at ${approval.url}`);
  const proof = await approval.waitForProof(decision.challenge.id);
  const result = await gateway.execute(action, runTool, { proofEnvelope: proof });
  console.log(result.ok, result.receipt.payload.id);
}
```

Deferred (not in v0.2): an optional `createPermitRail({ policy, approval, receiptKeyPair })`
sugar. v0.2 keeps the single, explicit `PermitRailGateway` API. Revisit only if the
verbose wiring proves annoying in practice.

## Server endpoints (localhost only)

- `GET /` approval page (HTML with inline CSS and JS, dark console look).
- `GET /api/pending` list pending challenges:
  `[{ id, tool, audience, subject, purpose, input, risk, createdAt, expiresAt }]`.
- `GET /api/challenge/:id` `{ status: 'pending' | 'approved' | 'denied' | 'expired' }`, the page polls this.
  `expired` is a derived API/page state only: the server returns it when `expiresAt` has passed.
  The core provider challenge status stays `pending | approved | denied` and is not changed in v0.2.
- `POST /api/approve/:id` body `{ approvedBy? }`, calls `provider.approve` (signs the proof), returns `{ ok: true }`.
- `POST /api/deny/:id` body `{ reason? }`, calls `provider.deny` (signs a receipt), returns `{ ok: true }`.

Bind to 127.0.0.1 only. No auth (local trust boundary). Reject non-localhost
requests. Accept small JSON bodies only.

## Await mechanic

- `waitForProof(id)` polls `provider.getChallenge(id)` on an interval (for example 400ms).
- Resolves with the signed proof envelope when status is approved.
- Rejects when status is denied, when the challenge passes `expiresAt`, or when `timeoutMs` elapses.
- No new crypto: `provider.approve` already signs the Ed25519 proof, this only waits for it.

## Page UX (dark console aesthetic)

- A queue of pending approvals, newest first, updated live by polling `/api/pending`.
- Each card shows: tool (CALL badge), audience, subject, purpose, the exact input PermitRail binds to the proof, and a risk pill.
- Buttons: Approve, Deny. On action the card shows the result, then clears.
- A banner: "Local approval. Single user, in memory, localhost only. For demos and internal tools, not production auth."

## Demo

`examples/local-approval-agent/`:
1. Policy: `payments.create_transfer` requires proof, input-hash bound.
2. `startLocalApproval()`, print the URL.
3. Simulated agent attempts a transfer.
4. `authorize` returns `require_proof`, print "approve at <url>".
5. Human approves in the browser.
6. `execute` runs once, print the signed receipt id.
7. Replay the same proof, show it refused.

## Tests (node:test, match existing style)

- `waitForProof` resolves with a verifiable proof after approve, `gateway.execute` succeeds, replay refused.
- `waitForProof` rejects after deny.
- `waitForProof` rejects after expiry or timeout.
- `/api/pending` lists the challenge, `/api/approve` signs, `/api/deny` seals a receipt.
- Server binds to 127.0.0.1, non-local requests rejected.
- The signed proof verifies with `verifyProof` against `approval.publicKeyPem`.

## Caveats (README and page)

Local approval is for demos, single-user dev, and internal tools. No page auth,
in-memory challenge store, localhost only. Not multi-user production auth.
Production approval uses your own channel (webhook today, Slack in v0.3) and a
durable store.
